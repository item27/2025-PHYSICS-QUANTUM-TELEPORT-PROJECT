# Системные требования (MVP)

## 1. Назначение
Описание обязательных требований к MVP Quantum Teleportation Visualizer. Используется разработчиками как источник истины при технических решениях.

## 2. Общие архитектурные принципы
- Frontend и backend независимы и могут запускаться отдельно.
- Frontend отвечает за UX, backend - за состояние и синхронизацию.
- Код читаемый, предсказуемый, без смешения UI и бизнес-логики.
- Документация на русском, ориентирована на людей без физики.

## 3. Режимы работы
Приложение обязательно поддерживает два режима:
- **Демонстрационный** - автономная пошаговая визуализация без сервера.
- **Сетевой** - работа с backend и WebSocket, сервер управляет состоянием и шагами.

## 4. Демонстрационный режим
- Не использует backend, все состояния локальные.
- Логика телепортации работает на frontend, без сетевых запросов.
- Показывает пошаговый процесс, позволяет запускать его повторно.
- Нет распределения ролей и синхронизации между пользователями.

## 5. Сетевой режим
- Требует backend и WebSocket для обмена в реальном времени.
- Каждый клиент представляет одну роль; сервер управляет текущим шагом и состояниями.
- Изменения транслируются всем подключённым клиентам.

## 6. UX требования
- Интерфейс понятен без знания физики: простой язык, подсказки, пошаговые действия.
- Ошибки объясняются и не пугают пользователя.
- Терминология аккуратная и узнаваемая для подготовленных пользователей.

## 7. Backend
- HTTP API описано в `api/openapi.yaml`.
- Все публичные типы и функции документированы.
- Логика разделена: управление сессиями, модель состояний, валидация, WebSocket-обновления.

## 8. Frontend
- Два режима в интерфейсе: демонстрационный и сетевой.
- Сценарий сетевого режима: подключение, выбор роли, получение токена, подключение к WebSocket, выполнение разрешённых действий, выход из роли.
- Демо-режим использует локальное состояние, переключается между шагами протокола и показывает ключевые события.

## 9. Нефункциональные требования
- Проект собирается и запускается локально и через Docker.
- Настройки (порты, адреса API и WS) передаются через переменные окружения.
- Код и тексты дружелюбны и объясняют «зачем», а не только «что».
