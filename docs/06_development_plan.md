# План разработки MVP

## 1. Цели этапа

* Собрать работающий демонстрационный режим.
* Поднять сетевой режим с базовыми ролями и шагами.
* Подготовить понятную документацию и сценарии запуска.

## 2. Приоритеты

1. Утвердить контракт API (`api/openapi.yaml`) и модель состояния.
2. Реализовать backend: управление сессиями, валидация шагов, WebSocket-рассылка.
3. Реализовать frontend демо-режим.
4. Реализовать frontend сетевого режима и интеграцию с backend.
5. Проверить совместную работу и обновить документацию.

## 3. Этапы

* **Аналитика**: сверить требования и архитектуру, уточнить UX сценарии.
* **Backend**: REST + WebSocket, память как хранилище, логирование основных событий.
* **Frontend демо**: шаги протокола, локальное состояние, подсказки и визуализации.
* **Frontend сетевой**: формы подключения, отображение ролей, работа с WebSocket, управление шагами роли.
* **Тестирование**: ручные проверки сценариев, запуск через Docker.

## 4. Риски и ограничения

* Нет постоянного хранилища: при рестарте сервера сессии теряются (приемлемо для MVP).
* Сетевой режим зависит от стабильности WebSocket; предусмотреть понятные сообщения об ошибках и переподключении.

## 5. Критерии готовности

* Оба режима запускаются из инструкции в README.
* API соответствует `api/openapi.yaml`.
* WebSocket обновляет клиентов при каждом действии.
* Демо-режим понятен без чтения кода.
* Документация отражает фактическое поведение приложения.
